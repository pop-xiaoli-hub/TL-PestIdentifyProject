# 适老化技术方案调研

> 植小保 iOS 项目 — 适老化实现路径讨论

---

## 背景

适老化的本质是：**同一个功能入口，根据用户模式展示不同风格的 UI**，但业务逻辑（网络请求、数据处理）只维护一份。

核心原则：**VC 层不变，只换 View 层的表现形式。**

---

## 方案一：Theme 主题系统（推荐基础方案）

定义一个全局 `TLTheme` 配置类，View 直接读取当前主题参数，不写任何 `if/else`。

```objc
@interface TLTheme : NSObject
@property CGFloat bodyFontSize;    // 普通: 14   适老: 20
@property CGFloat titleFontSize;   // 普通: 18   适老: 26
@property CGFloat cellHeight;      // 普通: 60   适老: 90
@property UIEdgeInsets padding;
+ (instancetype)current;           // 全局读取当前主题
@end
```

View 里直接使用：
```objc
self.titleLabel.font = [UIFont systemFontOfSize:[TLTheme current].titleFontSize];
```

切换适老化时只需更新 `TLTheme.current`，所有页面自动响应，**VC 和 View 完全不感知模式切换**。

**适用场景：** 适老差异主要集中在字号、间距、颜色（覆盖约 90% 的适老需求）。

---

## 方案二：同一 VC，切换两套 View（Protocol 隔离）

VC 业务逻辑只写一份，根据模式在初始化时选择加载哪套 View。

```objc
- (void)loadView {
    if ([UserSession shared].isElderMode) {
        self.view = [[ElderRecordListView alloc] init];
    } else {
        self.view = [[RecordListView alloc] init];
    }
}
```

两套 View 遵循同一个 Protocol，VC 不关心具体是哪个 View：

```objc
@protocol RecordListViewProtocol
@property (nonatomic, strong) NSArray *records;
@property (nonatomic, copy) void(^onItemTap)(TLPestRecord *);
@end
```

**适用场景：** 适老版与普通版布局差异极大，无法通过调整参数复用同一套布局。

---

## 方案三：Theme + View 继承组合（推荐完整方案）

结合方案一和方案二的优势：

- **Theme 处理共性差异**（字号、间距、颜色）→ 无需两套 View
- **继承处理布局差异**（局部页面布局大改）→ 子类只 override 需要改变的部分

```
RecordListView（基类，普通版，含完整业务绑定逻辑）
    └── ElderRecordListView（子类，只 override 布局方法）
```

**适用场景：** 大多数页面用 Theme 搞定，少数页面（如首页）布局变化较大时，局部用继承补充。

---

## 导航层（Router）如何配合

适老化不需要在 Router 层区分，Router 统一路由到同一个 VC，适老判断在 VC/Theme 内部处理：

```
Router → RecordListViewController
              ↓
         loadView 时读取 TLTheme.current / UserSession.isElderMode
              ↓
         加载对应 View（或由 Theme 参数驱动同一套 View）
```

---

## 方案对比

| 方案 | 代码量 | 灵活性 | 维护成本 | 适合场景 |
|------|--------|--------|----------|----------|
| Theme 系统 | 少 | 中 | 低 | 差异主要是字号/间距/颜色 |
| Protocol 切换 View | 多 | 高 | 高 | 两套 UI 差异极大 |
| View 继承 | 中 | 中 | 中 | 局部布局差异 |
| **Theme + 继承组合** | **中** | **最高** | **中** | **推荐，兼顾两种情况** |

---

## 结论建议

1. **先建立 `TLTheme` 系统**，作为项目基础规范，所有 View 从一开始就读取 Theme 参数
2. **普通页面**：Theme 驱动，无需额外工作
3. **布局差异大的页面**：在 Theme 基础上，用 View 继承处理剩余差异
4. **VC 层始终只有一份**，业务逻辑不重复
